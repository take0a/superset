# Superset の設定

## superset_config.py

Superset は、[config.py モジュール](https://github.com/apache/superset/blob/master/superset/config.py) を通じて、数百もの構成可能なパラメータを公開しています。公開されている変数とオブジェクトは、構成、変更、インターフェイスする可能性のあるほとんどのもののパブリック インターフェイスとして機能します。この Python モジュールには、これらすべてのパラメータ、適切なデフォルト、およびコメント形式の豊富なドキュメントが含まれています。

アプリケーションを構成するには、独自の構成モジュールを作成する必要があります。これにより、これらのパラメータの一部または多くをオーバーライドできます。コア モジュールを変更する代わりに、独自のモジュール (通常は `superset_config.py` という名前のファイル) を定義します。

このファイルを `PYTHONPATH` に追加するか、`superset_config.py` のフル パスを指定する環境変数 `SUPERSET_CONFIG_PATH` を作成します。

たとえば、`superset_config.py` が `/app` ディレクトリの下にある Linux ベースのシステムで Superset を直接デプロイする場合は、次を実行できます。

```bash
export SUPERSET_CONFIG_PATH=/app/superset_config.py
```

公式の Superset イメージをベース イメージとして独自のカスタム Dockerfile を使用している場合は、次のようにオーバーライドを追加できます。

```bash
COPY --chown=superset superset_config.py /app/
ENV SUPERSET_CONFIG_PATH /app/superset_config.py
```

Docker Compose デプロイメントでは、特定の規則を使用してアプリケーション構成を異なる方法で処理します。
詳細については、[docker Compose のヒントと構成](/docs/installation/docker-compose#docker-compose-tips--configuration) を参照してください。

以下は、`superset_config.py` ファイルで設定できるパラメーターのほんの一例です:

```
# Superset specific config
ROW_LIMIT = 5000

# Flask App Builder configuration
# Your App secret key will be used for securely signing the session cookie
# and encrypting sensitive information on the database
# Make sure you are changing this key for your deployment with a strong key.
# Alternatively you can set it with `SUPERSET_SECRET_KEY` environment variable.
# You MUST set this for production environments or the server will refuse
# to start and you will see an error in the logs accordingly.
SECRET_KEY = 'YOUR_OWN_RANDOM_GENERATED_SECRET_KEY'

# The SQLAlchemy connection string to your database backend
# This connection defines the path to the database that stores your
# superset metadata (slices, connections, tables, dashboards, ...).
# Note that the connection information to connect to the datasources
# you want to explore are managed directly in the web UI
# The check_same_thread=false property ensures the sqlite client does not attempt
# to enforce single-threaded access, which may be problematic in some edge cases
SQLALCHEMY_DATABASE_URI = 'sqlite:////path/to/superset.db?check_same_thread=false'

# Flask-WTF flag for CSRF
WTF_CSRF_ENABLED = True
# Add endpoints that need to be exempt from CSRF protection
WTF_CSRF_EXEMPT_LIST = []
# A CSRF token that expires in 1 year
WTF_CSRF_TIME_LIMIT = 60 * 60 * 24 * 365

# Set this API key to enable Mapbox visualizations
MAPBOX_API_KEY = ''
```

:::tip
通常は、コアの [superset/config.py](https://github.com/apache/superset/blob/master/superset/config.py) から変更したい部分のみを、関連するコメントとともにコピーして、独自の `superset_config.py` ファイルに貼り付けることに注意してください。
:::

[superset/config.py](https://github.com/apache/superset/blob/master/superset/config.py) で定義されているすべてのパラメータとデフォルト値は、ローカルの `superset_config.py` で変更できます。管理者は、ローカルで構成できる内容と、設定されているデフォルト値を理解するために、このファイルを読む必要があります。

`superset_config.py` は Flask 構成モジュールとして機能するため、Flask 自体の設定だけでなく、Superset がバンドルする `flask-wtf`、`flask-caching`、`flask-migrate`、`flask-appbuilder` などの Flask 拡張機能の設定も変更できます。これらの拡張機能はそれぞれ、複雑な構成が可能です。

Superset が使用する Web フレームワークである Flask App Builder にも、多くの構成設定が用意されています。設定方法の詳細については、[Flask App Builder ドキュメント](https://flask-appbuilder.readthedocs.org/en/latest/config.html) を参照してください。

少なくとも、`SECRET_KEY` と `SQLALCHEMY_DATABASE_URI` を変更する必要があります。それぞれの詳細については、引き続きお読みください。

## SECRET_KEYの指定

### 初期SECRET_KEYの追加

Superset を起動するには、ユーザーが指定した SECRET_KEY が必要です。この要件は、[安全な構成を強制するためにバージョン 2.1.0 で追加されました](https://preset.io/blog/superset-security-update-default-secret_key-vulnerability/)。次のように、強力な SECRET_KEY を `superset_config.py` ファイルに追加します:

```python
SECRET_KEY = 'YOUR_OWN_RANDOM_GENERATED_SECRET_KEY'
```

`openssl rand -base64 42` を使用して強力で安全なキーを生成できます。

:::caution 強力な秘密鍵を使用する
このキーは、セッション Cookie に安全に署名し、Superset のアプリケーション メタデータ データベースに保存されている機密情報を暗号化するために使用されます。
展開では、複雑で一意のキーを使用する必要があります。
:::

### 新しい SECRET_KEY へのローテーション

既存の SECRET_KEY を変更する場合は、既存の SECRET_KEY を `superset_config.py` ファイルに `PREVIOUS_SECRET_KEY =` として追加し、新しいキーを `SECRET_KEY =` として指定します。現在の SECRET_KEY は次のコマンドで確認できます。Superset を Docker で実行している場合は、Superset アプリケーション コンテナー内から実行します:

```python
superset shell
from flask import current_app; print(current_app.config["SECRET_KEY"])
```

これらの値を使用して `superset_config.py` を保存し、`superset re-encrypt-secrets` を実行します。

## 運用メタデータデータベースの設定

Superset には、チャートの定義、ダッシュボード、その他多くのものなど、管理する情報を格納するためのデータベースが必要です。

デフォルトでは、Superset は [SQLite](https://www.sqlite.org/) を使用するように構成されています。これは、インストールを必要とせずに簡単にすばやく開始できる自己完結型の単一ファイル データベースです。ただし、実稼働環境では、セキュリティ、スケーラビリティ、およびデータ整合性の理由から、SQLite の使用は強く推奨されません。
サポートされているデータベース エンジンのみを使用し、別のホストまたはコンテナーで別のデータベース エンジンを使用することを検討することが重要です。

Superset は次のデータベース エンジン/バージョンをサポートしています:

| Database Engine                           | Supported Versions                       |
| ----------------------------------------- | ---------------------------------------- |
| [PostgreSQL](https://www.postgresql.org/) | 10.X, 11.X, 12.X, 13.X, 14.X, 15.X, 16.X |
| [MySQL](https://www.mysql.com/)           | 5.7, 8.X                                 |

次のデータベース ドライバーと接続文字列を使用します:

| Database                                  | PyPI package              | Connection String                                                      |
| ----------------------------------------- | ------------------------- | ---------------------------------------------------------------------- |
| [PostgreSQL](https://www.postgresql.org/) | `pip install psycopg2`    | `postgresql://<UserName>:<DBPassword>@<Database Host>/<Database Name>` |
| [MySQL](https://www.mysql.com/)           | `pip install mysqlclient` | `mysql://<UserName>:<DBPassword>@<Database Host>/<Database Name>`      |

:::tip
メタデータ ストアを適切に設定する方法については、このドキュメントでは説明しません。サービスとサポート インフラストラクチャおよびバックアップ戦略を処理するには、[Amazon RDS](https://aws.amazon.com/rds/) や [Google Cloud Databases](https://cloud.google.com/products/databases?hl=en) などのホスト型マネージド サービスを使用することをお勧めします。
:::

Superset メタストアを構成するには、`superset_config` の `SQLALCHEMY_DATABASE_URI` 構成キーを適切な接続文字列に設定します。

## WSGI HTTP サーバー上で実行

Superset は NGINX または Apache で実行できますが、非同期モードで Gunicorn を使用することをお勧めします。これにより、優れた同時実行性を実現でき、インストールと構成も非常に簡単です。この Flask WSGI アプリケーションを自分の環境で適切に動作するように設定するには、好みのテクノロジのドキュメントを参照してください。以下は、本番環境で適切に動作することがわかっている非同期設定です:

```
      -w 10 \
      -k gevent \
      --worker-connections 1000 \
      --timeout 120 \
      -b  0.0.0.0:6666 \
      --limit-request-line 0 \
      --limit-request-field_size 0 \
      --statsd-host localhost:8125 \
      "superset.app:create_app()"
```

詳細については、[Gunicorn のドキュメント](https://docs.gunicorn.org/en/stable/design.html)を参照してください。_開発用 Web サーバー (`superset run` または `flask run`) は、本番環境での使用を目的としたものではないことに注意してください。_

Gunicorn を使用していない場合は、`superset_config.py` で `COMPRESS_REGISTER = False` を設定して `flask-compress` の使用を無効にすることをお勧めします。

現在、Google BigQuery Python SDK は、`gevent` による Python コア ライブラリへの動的なモンキーパッチのため、`gevent` と互換性がありません。
そのため、Superset で `BigQuery` データソースを使用する場合は、`gevent` 以外の `gunicorn` ワーカー タイプを使用する必要があります。

## HTTPS 構成

ロードバランサーまたはリバースプロキシ (nginx など) を介して HTTPS アップストリームを構成し、トラフィックが Superset アプリケーションに到達する前に SSL/TLS オフロードを行うことができます。この設定では、アラートとレポートのチャートのスナップショットを作成する Celery ワーカーからのローカルトラフィックは、イングレスポイントの背後から `http://` URL で Superset にアクセスできます。
公式の Superset Docker イメージを使用している場合は、[Gunicorn で SSL](https://docs.gunicorn.org/en/stable/settings.html#ssl) (Python Web サーバー) を構成することもできます。

## ロードバランサーの背後の構成

ロードバランサーまたはリバースプロキシ (AWS 上の NGINX や ELB など) の背後で superset を実行している場合は、ロードバランサーが superset インスタンスが実行中であるかどうかを認識できるように、ヘルスチェックエンドポイントを利用する必要がある場合があります。これは `/health` で提供され、Web サーバーが実行中の場合は「OK」を含む 200 応答を返します。

ロード バランサが `X-Forwarded-For/X-Forwarded-Proto` ヘッダーを挿入している場合は、ヘッダーを抽出して使用するには、superset 構成ファイル (`superset_config.py`) で `ENABLE_PROXY_FIX = True` を設定する必要があります。

SSL 暗号化を提供するためにリバース プロキシを使用する場合は、`X-Forwarded-Proto` の明示的な定義が必要になることがあります。Apache Web サーバーの場合は、次のように設定できます:

```
RequestHeader set X-Forwarded-Proto "https"
```

## カスタム OAuth2 構成

Superset は Flask-AppBuilder (FAB) 上に構築されており、多くのプロバイダー (GitHub、Twitter、LinkedIn、Google、Azure など) をすぐにサポートします。さらに、Superset は、「コード」認証をサポートする他の OAuth2 認証サーバー実装に接続するように構成できます。

pip パッケージ [`Authlib`](https://authlib.org/) が Web サーバーにインストールされていることを確認してください。

まず、Superset `superset_config.py` で認証を設定します。

```python
from flask_appbuilder.security.manager import AUTH_OAUTH

# Set the authentication type to OAuth
AUTH_TYPE = AUTH_OAUTH

OAUTH_PROVIDERS = [
    {   'name':'egaSSO',
        'token_key':'access_token', # Name of the token in the response of access_token_url
        'icon':'fa-address-card',   # Icon for the provider
        'remote_app': {
            'client_id':'myClientId',  # Client Id (Identify Superset application)
            'client_secret':'MySecret', # Secret for this Client Id (Identify Superset application)
            'client_kwargs':{
                'scope': 'read'               # Scope for the Authorization
            },
            'access_token_method':'POST',    # HTTP Method to call access_token_url
            'access_token_params':{        # Additional parameters for calls to access_token_url
                'client_id':'myClientId'
            },
            'jwks_uri':'https://myAuthorizationServe/adfs/discovery/keys', # may be required to generate token
            'access_token_headers':{    # Additional headers for calls to access_token_url
                'Authorization': 'Basic Base64EncodedClientIdAndSecret'
            },
            'api_base_url':'https://myAuthorizationServer/oauth2AuthorizationServer/',
            'access_token_url':'https://myAuthorizationServer/oauth2AuthorizationServer/token',
            'authorize_url':'https://myAuthorizationServer/oauth2AuthorizationServer/authorize'
        }
    }
]

# Will allow user self registration, allowing to create Flask users from Authorized User
AUTH_USER_REGISTRATION = True

# The default user self registration role
AUTH_USER_REGISTRATION_ROLE = "Public"
```

次に、`SupersetSecurityManager` を拡張し、`oauth_user_info` をオーバーライドする `CustomSsoSecurityManager` を作成します:

```python
import logging
from superset.security import SupersetSecurityManager

class CustomSsoSecurityManager(SupersetSecurityManager):

    def oauth_user_info(self, provider, response=None):
        logging.debug("Oauth2 provider: {0}.".format(provider))
        if provider == 'egaSSO':
            # As example, this line request a GET to base_url + '/' + userDetails with Bearer  Authentication,
    # and expects that authorization server checks the token, and response with user details
            me = self.appbuilder.sm.oauth_remotes[provider].get('userDetails').data
            logging.debug("user_data: {0}".format(me))
            return { 'name' : me['name'], 'email' : me['email'], 'id' : me['user_name'], 'username' : me['user_name'], 'first_name':'', 'last_name':''}
    ...
```

このファイルは、`custom_sso_security_manager.py` という名前で、`superset_config.py` と同じディレクトリに配置する必要があります。最後に、`superset_config.py` に次の 2 行を追加します:

```
from custom_sso_security_manager import CustomSsoSecurityManager
CUSTOM_SECURITY_MANAGER = CustomSsoSecurityManager
```

**Notes**

- 必要に応じて OAuth2 認証プロバイダーを構成する場合、リダイレクト URL は `https://<superset-webserver>/oauth-authorized/<provider-name>` になります。たとえば、上記の構成の場合、リダイレクト URL は `https://<superset-webserver>/oauth-authorized/egaSSO` になります。

- OAuth2 認証サーバーが OpenID Connect 1.0 をサポートしている場合は、`api_base_url`、`access_token_url`、`authorize_url` およびユーザー情報エンドポイント、jwks uri などのその他の必要なオプションを指定せずに、その構成ドキュメント URL のみを構成できます。たとえば、次のようになります。

  ```python
  OAUTH_PROVIDERS = [
    {   'name':'egaSSO',
        'token_key':'access_token', # Name of the token in the response of access_token_url
        'icon':'fa-address-card',   # Icon for the provider
        'remote_app': {
            'client_id':'myClientId',  # Client Id (Identify Superset application)
            'client_secret':'MySecret', # Secret for this Client Id (Identify Superset application)
            'server_metadata_url': 'https://myAuthorizationServer/.well-known/openid-configuration'
        }
    }
  ]
  ```

### Flask-OIDC を使用した Keycloak 固有の設定

Keycloak を OpenID Connect 1.0 プロバイダーとして使用している場合、[`Authlib`](https://authlib.org/) に基づく上記の構成は機能しない可能性があります。この場合、[`Flask-OIDC`](https://pypi.org/project/flask-oidc/) を使用するのが実行可能なオプションです。

pip パッケージ [`Flask-OIDC`](https://pypi.org/project/flask-oidc/) が Web サーバーにインストールされていることを確認してください。これはバージョン 2.2.0 を使用して正常にテストされました。このパッケージには、依存関係として [`Flask-OpenID`](https://pypi.org/project/Flask-OpenID/) が必要です。

次のコードは、新しいセキュリティ マネージャーを定義します。これを、`superset_config.py` ファイルと同じディレクトリにある `keycloak_security_manager.py` という名前の新しいファイルに追加します。

```python
from flask_appbuilder.security.manager import AUTH_OID
from superset.security import SupersetSecurityManager
from flask_oidc import OpenIDConnect
from flask_appbuilder.security.views import AuthOIDView
from flask_login import login_user
from urllib.parse import quote
from flask_appbuilder.views import ModelView, SimpleFormView, expose
from flask import (
    redirect,
    request
)
import logging

class OIDCSecurityManager(SupersetSecurityManager):

    def __init__(self, appbuilder):
        super(OIDCSecurityManager, self).__init__(appbuilder)
        if self.auth_type == AUTH_OID:
            self.oid = OpenIDConnect(self.appbuilder.get_app)
        self.authoidview = AuthOIDCView

class AuthOIDCView(AuthOIDView):

    @expose('/login/', methods=['GET', 'POST'])
    def login(self, flag=True):
        sm = self.appbuilder.sm
        oidc = sm.oid

        @self.appbuilder.sm.oid.require_login
        def handle_login():
            user = sm.auth_user_oid(oidc.user_getfield('email'))

            if user is None:
                info = oidc.user_getinfo(['preferred_username', 'given_name', 'family_name', 'email'])
                user = sm.add_user(info.get('preferred_username'), info.get('given_name'), info.get('family_name'),
                                   info.get('email'), sm.find_role('Gamma'))

            login_user(user, remember=False)
            return redirect(self.appbuilder.get_url_for_index)

        return handle_login()

    @expose('/logout/', methods=['GET', 'POST'])
    def logout(self):
        oidc = self.appbuilder.sm.oid

        oidc.logout()
        super(AuthOIDCView, self).logout()
        redirect_url = request.url_root.strip('/') + self.appbuilder.get_url_for_login

        return redirect(
            oidc.client_secrets.get('issuer') + '/protocol/openid-connect/logout?redirect_uri=' + quote(redirect_url))
```

次に、`superset_config.py` ファイルに以下を追加します:

```python
from keycloak_security_manager import OIDCSecurityManager
from flask_appbuilder.security.manager import AUTH_OID, AUTH_REMOTE_USER, AUTH_DB, AUTH_LDAP, AUTH_OAUTH
import os

AUTH_TYPE = AUTH_OID
SECRET_KEY: 'SomethingNotEntirelySecret'
OIDC_CLIENT_SECRETS =  '/path/to/client_secret.json'
OIDC_ID_TOKEN_COOKIE_SECURE = False
OIDC_OPENID_REALM: '<myRealm>'
OIDC_INTROSPECTION_AUTH_METHOD: 'client_secret_post'
CUSTOM_SECURITY_MANAGER = OIDCSecurityManager

# Will allow user self registration, allowing to create Flask users from Authorized User
AUTH_USER_REGISTRATION = True

# The default user self registration role
AUTH_USER_REGISTRATION_ROLE = 'Public'
```

クライアント固有の OpenID 情報を `client_secret.json` というファイルに保存します。このファイルを `superset_config.py` と同じディレクトリに作成します:

```json
{
    "<myOpenIDProvider>": {
        "issuer": "https://<myKeycloakDomain>/realms/<myRealm>",
        "auth_uri": "https://<myKeycloakDomain>/realms/<myRealm>/protocol/openid-connect/auth",
        "client_id": "https://<myKeycloakDomain>",
        "client_secret": "<myClientSecret>",
        "redirect_uris": [
            "https://<SupersetWebserver>/oauth-authorized/<myOpenIDProvider>"
  ],
        "userinfo_uri": "https://<myKeycloakDomain>/realms/<myRealm>/protocol/openid-connect/userinfo",
        "token_uri": "https://<myKeycloakDomain>/realms/<myRealm>/protocol/openid-connect/token",
        "token_introspection_uri": "https://<myKeycloakDomain>/realms/<myRealm>/protocol/openid-connect/token/introspect"
  }
}
```

## LDAP認証

FAB は、LDAP サーバーに対するユーザー資格情報の認証をサポートしています。
LDAP を使用するには、[python-ldap](https://www.python-ldap.org/en/latest/installing.html) パッケージをインストールする必要があります。
詳細については、[FAB の LDAP ドキュメント](https://flask-appbuilder.readthedocs.io/en/latest/security.html#authentication-ldap) を参照してください。

## LDAP または OAUTH グループを Superset ロールにマッピングする

Flask-AppBuilder の AUTH_ROLES_MAPPING は、LDAP/OAUTH グループ名から FAB ロールにマッピングする辞書です。
これは、LDAP または OAuth を使用して認証するユーザーにロールを割り当てるために使用されます。

### OAUTH グループを Superset ロールにマッピングする

次の `AUTH_ROLES_MAPPING` ディクショナリは、OAUTH グループ "superset_users" を Superset ロール "Gamma" と "Alpha" にマッピングし、OAUTH グループ "superset_admins" を Superset  ロール "Admin" にマッピングします。

```python
AUTH_ROLES_MAPPING = {
"superset_users": ["Gamma","Alpha"],
"superset_admins": ["Admin"],
}
```

### LDAP グループを Superset ロールにマッピングする

次の `AUTH_ROLES_MAPPING` 辞書は、LDAP DN "cn=superset_users,ou=groups,dc=example,dc=com" を Superset ロール "Gamma" と "Alpha" にマッピングし、LDAP DN "cn=superset_admins,ou=groups,dc=example,dc=com" を Superset ロール "Admin" にマッピングします。

```python
AUTH_ROLES_MAPPING = {
"cn=superset_users,ou=groups,dc=example,dc=com": ["Gamma","Alpha"],
"cn=superset_admins,ou=groups,dc=example,dc=com": ["Admin"],
}
```

注意: これには `AUTH_LDAP_SEARCH` を設定する必要があります。詳細については、[FAB セキュリティ ドキュメント](https://flask-appbuilder.readthedocs.io/en/latest/security.html)を参照してください。

### ログイン時にロールを同期する

また、`AUTH_ROLES_SYNC_AT_LOGIN` 構成変数を使用して、Flask-AppBuilder がユーザーのロールを LDAP/OAUTH グループと同期する頻度を制御することもできます。`AUTH_ROLES_SYNC_AT_LOGIN` が True に設定されている場合、Flask-AppBuilder はユーザーがログインするたびにそのロールを同期します。`AUTH_ROLES_SYNC_AT_LOGIN` が False に設定されている場合、Flask-AppBuilder はユーザーが最初に登録したときにのみそのロールを同期します。

## Flask アプリ構成フック

`FLASK_APP_MUTATOR` は、環境に提供できる設定関数で、アプリ オブジェクトを受け取って、任意の方法で変更できます。たとえば、`FLASK_APP_MUTATOR` を `superset_config.py` に追加して、セッション クッキーの有効期限を 24 時間に設定します:

```python
from flask import session
from flask import Flask


def make_session_permanent():
    '''
    Enable maxAge for the cookie 'session'
    '''
    session.permanent = True

# Set up max age of session to 24 hours
PERMANENT_SESSION_LIFETIME = timedelta(hours=24)
def FLASK_APP_MUTATOR(app: Flask) -> None:
    app.before_request_funcs.setdefault(None, []).append(make_session_permanent)
```

## 機能フラグ

多様なユーザーをサポートするために、Superset にはデフォルトで有効になっていない機能がいくつかあります。たとえば、一部のユーザーにはより強力なセキュリティ制限が適用されますが、他のユーザーには適用されない場合があります。そのため、Superset では、ユーザーが構成によって一部の機能を有効または無効にすることができます。機能の所有者は、Superset にオプションの機能を追加できますが、影響を受けるのは一部のユーザーのみです。

`superset_config.py` のフラグを使用して機能を有効または無効にできます:

```python
FEATURE_FLAGS = {
    'PRESTO_EXPAND_DATA': False,
}
```

機能フラグの現在のリストは、[RESOURCES/FEATURE_FLAGS.md](https://github.com/apache/superset/blob/master/RESOURCES/FEATURE_FLAGS.md) にあります。
